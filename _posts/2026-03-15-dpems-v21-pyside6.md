## Phase 1: 데스크탑 앱 완성

설계를 마쳤으니 이제 실제로 돌아가는 앱을 만들 차례다. UI 프레임워크로 **PySide6**를 선택했다. Qt 기반이라 시그널/슬롯으로 컴포넌트 간 통신이 깔끔하고, 나중에 Android 포팅 가능성도 염두에 뒀다.

## 엔진: 가중치 계산 루프

`engine.py`는 `QTimer`로 1초마다 틱을 발생시키고, 모든 태스크의 가중치를 재계산해 시그널로 전달한다.

### 가중치 수식

경과 시간 `t`, 목표 시간 `T` (100점), 임계 시간 `C` (200점)를 기준으로 4가지 곡선 타입을 지원한다.

**LINEAR_POINT** (기본)
```
t <= T:  score = (t/T) × 100
t >  T:  score = 100 + ((t-T)/(C-T)) × 100
```

**EXP** (지수 가속)
```
score = 100 + (e^(α×t/C) - e^(α×T/C)) / (e^α - e^(α×T/C)) × 100
```

최종 점수는 `score × base_priority`로, 중요도에 따라 배율이 적용된다.

### 정렬 기준

단순 점수 내림차순이 아니다. 3단계 tier로 먼저 분류한다.

```
tier 0: weight > 0 (활성)
tier 1: weight == 0 이지만 오늘 일정
tier 2: 오늘 일정 아님 (최하단)
```

같은 tier 안에서는 weight 내림차순, 동점이면 마감 임박 순으로 정렬된다.

## 시스템 상태 판정

상단 전광판에 현재 시스템 상태를 5단계로 표시한다.

| 상태 | 조건 |
|------|------|
| CRITICAL | 최상위 태스크 가중치 ≥ 200 |
| CONFLICT | 토글 ON 태스크가 둘 이상이고 가중치 ≥ 100 |
| RUNNING | 최상위 태스크 토글 ON |
| TARGET | 최상위 가중치 ≥ 100 |
| STABLE | 그 외 |

## 다이얼로그 설계 원칙

태스크 추가/수정 다이얼로그에서 중요한 결정이 있었다.

`exec()` 대신 `show()` + `accepted.connect()` 방식을 쓴다. `exec()`는 이벤트 루프를 블로킹하기 때문에 다이얼로그가 열린 상태에서 dashboard를 드래그하면 멈춰버린다. `show()` 방식이면 백그라운드에서 dashboard가 계속 반응한다.

또한 `super().__init__(None)`으로 부모 없이 생성해 부모 창의 스타일이 다이얼로그에 상속되는 것을 막았다.

## TEST MODE

개발하면서 "6시간 후 상태를 지금 확인하고 싶다"는 상황이 자주 생겼다. TEST MODE를 만들어 해결했다.

- 시간 오프셋: +10분, +1시간, +1일 버튼
- 속도 배율: 슬라이더로 타이머 간격 축소
- TEST_MODE 종료 시 오프셋/속도 전부 초기화

`engine.py`는 `datetime.now()` 직접 호출 금지, 반드시 `self._get_now()`를 통해 호출한다. 이 메서드가 TEST_MODE일 때 오프셋을 적용한다.
